# 产品需求文档（PRD）—— YOLO 检测平台性能优化

---

## 一、项目信息

| 项 | 说明 |
|---|---|
| **项目名称** | `yolo_detection_platform`（性能优化专项） |
| **语言** | 中文 |
| **编程语言** | Python FastAPI + Vue 3 + TypeScript |
| **原始需求复述** | 对现有 YOLO 检测平台进行系统性性能优化，方向涵盖：YOLO 推理加速、前后端缓存体系建设、大视频分片流式处理。 |

---

## 二、产品定义

### 2.1 优化目标

| # | 目标 | 量化指标 | 当前基线 | 目标值 |
|---|---|---|---|---|
| G1 | 大幅降低 YOLO 推理延迟 | 单张图片推理耗时 | ~800-1200ms (CPU) / ~200-400ms (GPU) | **< 300ms (CPU)** / **< 80ms (GPU)** |
| G2 | 构建系统化缓存层，减少重复计算 | 相同输入二次响应时间 | = 首次时间（无缓存） | **< 50ms（命中缓存）** |
| G3 | 保障大视频处理的稳定性与效率 | 支持的最大视频时长 | 受限于可用内存（~10min 1080p） | **支持 >60min 4K 视频，内存峰值 < 500MB** |
| G4 | 提升系统并发处理能力 | 并发检测任务吞吐量 | 同一模型串行，约 1-2 task/s | **≥ 5 task/s（批处理 + 流水线）** |

### 2.2 用户故事

| # | 用户故事 | 当前痛点 | 期望体验 |
|---|---|---|---|
| US1 | 作为**研究人员**，我希望快速迭代 YOLO 模型验证，对同一张图片用不同参数反复测试 | 每次检测都要完整走推理流程（~2s），改个置信度阈值也要重新跑一遍 | 首次推理后结果缓存在 Redis，切换参数秒级返回 |
| US2 | 作为**运维人员**，我上传了一段 30 分钟的监控视频进行目标检测 | 视频上传后服务器内存飙升、进度卡死、页面刷新后丢失状态 | 分片上传有进度条、后端流式处理内存稳定、刷新页面可恢复任务 |
| US3 | 作为**普通用户**，我在 YOLO+LLM 协同模式下检测一张复杂场景图片 | YOLO 推理 + 每个区域串行调 LLM，等待超过 15 秒 | 区域分析并行度提升，总耗时 < 5 秒 |
| US4 | 作为**前端开发者**，我注意到用户频繁切换历史记录列表页面 | 每次进入历史页都重新请求全部数据，列表加载 2-3 秒 | 前端缓存 + 后端分页缓存，列表秒开 |
| US5 | 作为**系统管理员**，我部署了 GPU 服务器但感觉利用率很低 | 模型每次加载/卸载开销大，GPU 大部分时间空闲 | 模型常驻显存 + 批处理推理，GPU 利用率 > 60% |

## 三、技术规范

### 3.1 需求池

#### P0 —— 必须实现（本轮优化核心）

| 编号 | 需求 | 方向 | 说明 |
|---|---|---|---|
| P0-1 | YOLO 模型 ONNX/TensorRT 导出与推理 | 推理加速 | 将 .pt 模型导出为 ONNX 格式，GPU 环境进一步导出 TensorRT engine；推理时优先使用优化模型 |
| P0-2 | 同模型批处理推理 | 推理加速 | 短时间窗口内（100ms）的检测请求合并为 batch，单次前向传播处理多张图片 |
| P0-3 | 检测结果 Redis 缓存 | 缓存 | 对相同图片 + 相同模型 + 相同参数的检测结果做 hash 缓存（TTL 1h），命中后直接返回 |
| P0-4 | 视频分片上传 | 大视频处理 | 前端将大视频切为 10MB 分片上传，后端合并，支持断点续传和进度展示 |
| P0-5 | 视频流式逐帧处理（不落盘全量临时帧） | 大视频处理 | 当前方案一次性 extract_frames 落盘所有帧再处理；改为边读边处理，内存中仅保留当前帧 |

#### P1 —— 应该实现（性能显著提升）

| 编号 | 需求 | 方向 | 说明 |
|---|---|---|---|
| P1-1 | LLM 分析结果缓存 | 缓存 | 对相同区域裁剪图 + 相同 prompt 的 LLM 分析结果做语义 hash 缓存（TTL 24h），避免重复调用昂贵的 LLM API |
| P1-2 | 模型常驻显存 + 预热 | 推理加速 | 服务启动时预加载默认模型到 GPU 显存，设置 keep-alive 不因 LRU 淘汰，消除冷启动延迟 |
| P1-3 | API 响应 CDN/Edge 缓存 | 缓存 | 静态资源（标注图片、缩略图）通过 `Cache-Control` + ETag 缓存；历史记录列表响应增加 `If-None-Match` 支持 |
| P1-4 | 视频 GPU 加速编解码 | 推理加速 | 使用 GPU NVENC/NVDEC 替代 CPU 编解码，帧提取和标注视频合成在 GPU 侧完成 |
| P1-5 | 前端 Pinia 持久化 + 请求去重 | 缓存 | 检测配置、LLM 配置变更不频繁的数据前端缓存 5min；相同请求未返回时不重复发起 |

#### P2 —— 可以实现（长期优化）

| 编号 | 需求 | 方向 | 说明 |
|---|---|---|---|
| P2-1 | 模型量化推理（INT8/FP16） | 推理加速 | 对 ONNX 模型做 INT8 量化，精度损失 < 2%，推理速度再提升 2-4x |
| P2-2 | 视频自适应帧间隔 | 大视频处理 | 根据视频内容变化程度动态调整检测帧间隔（静态场景跳帧更多，动态场景密集检测） |
| P2-3 | 分布式任务队列 | 大视频处理 | 超长视频拆分为多段，分发到多个 worker 节点并行处理，最后合并结果 |
| P2-4 | 智能预加载 | 缓存 | 根据用户历史行为预测下一次检测的模型，后台预加载到内存/显存 |

### 3.2 简要方案建议

#### 方向一：YOLO 推理加速

**核心思路：模型优化 + 批处理 + 显存常驻**

```
当前：每次请求 → load_model (thread) → detect (serial, per-model lock) → return
优化：启动预热 → 模型常驻显存 → batch collect (100ms window) → ONNX/TensorRT 推理 → return
```

**关键技术点**：
1. **ONNX 导出**：`model.export(format="onnx")` → `onnxruntime` 推理，CPU 场景提速 2-3x
2. **TensorRT**：GPU 场景导出 TensorRT engine，FP16 推理提速 3-6x
3. **批处理**：使用 `asyncio.Queue` 收集 100ms 窗口内的请求，合并为 batch tensor 一次性推理
4. **模型预热**：启动时 `warm_up()` 跑一次空推理，触发 CUDA kernel 编译缓存

#### 方向二：前后端缓存

**核心思路：分层缓存（前端 → API → Redis → 数据库）**

```
前端 (Pinia + localStorage)
  ↓ miss
API 层 (响应 Cache-Control / ETag)
  ↓ miss
Redis 缓存层 (检测结果 / LLM 分析 / session)
  ↓ miss
MySQL 持久层
```

**关键技术点**：
1. **检测结果缓存**：key = `det:{sha256(image+model+params)}`，value = bbox JSON，TTL = 1h
2. **LLM 分析缓存**：key = `llm:{sha256(region_image+prompt)}`，TTL = 24h（昂贵调用优先缓存）
3. **API 响应缓存**：`Cache-Control: max-age=300` 用于缩略图/模型列表等低频变更接口
4. **前端请求去重**：Pinia store 中标记 inflight 请求，相同请求等待已有 Promise

#### 方向三：大视频分片处理

**核心思路：分片上传 + 流式读取 + 内存可控**

```
前端：10MB 分片上传 → 后端合并 → 流式逐帧处理
                                  ↓
            cv2.VideoCapture (stream) → read() → detect → write annotated frame
            (不 extract_frames 落盘)    ↑ 内存仅保留当前帧
```

**关键技术点**：
1. **分片上传**：前端 `File.slice()` 切割，并发 3 片上传，Redis 记录分片状态（支持断点续传）
2. **流式逐帧处理**：不再 `extract_frames` 全量落盘，改为 `cap.read()` → `yolo.detect(bytes)` → `out.write()` 一条流水线
3. **内存控制**：通过 `cv2.VideoCapture` 流式读取，内存峰值 ≈ 单帧大小 × 2（解码帧 + 标注帧），独立于视频时长
4. **断点恢复**：每处理 100 帧更新 Redis `video:task:{id}:last_frame`，中断后可从断点继续

### 3.4 待确认问题

| # | 问题 | 影响范围 | 建议 |
|---|---|---|---|
| Q1 | ONNX/TensorRT 是作为默认推理路径还是可选？如果用户未安装 onnxruntime 是否回退 ultralytics？ | 部署复杂度、兼容性 | **建议**: 自动检测环境，GPU 优先 TensorRT，CPU 优先 ONNX，均不可用回退 ultralytics 原生推理 |
| Q2 | 模型常驻显存会占用 GPU 资源，最多允许同时常驻几个模型？ | GPU 显存规划 | **建议**: 默认 1 个常驻（正在使用的模型），最大可配置。用户切换模型时先加载新模型再卸载旧模型 |
| Q3 | 检测结果缓存一致性：当用户更新自定义模型文件后，旧缓存是否应该批量失效？ | 缓存策略 | **建议**: 模型文件路径变更时，按模型 ID 批量清除相关缓存 key（`det:{model_id}:*`） |
| Q4 | 批处理窗口 100ms 会增加单次请求延迟，可接受的额外等待上限是多少？ | 用户体验、批处理效率权衡 | **建议**: 窗口 100ms 且最多合并 8 张图，超过任一限制立即推理。用户感知延迟增加 < 100ms |
| Q5 | 视频断点续传的分片存在哪里？是否需要定期清理未完成的分片？ | 存储管理 | **建议**: 分片存 `uploads/chunks/{user_id}/{task_id}/`，Redis 记录状态。24 小时未完成的任务自动清理 |
| Q6 | LLM 分析缓存是否考虑跨用户共享？（如两个用户用同一 API key 分析同一区域） | 成本节省 vs 数据隔离 | **建议**: 默认不跨用户（安全优先），管理员可开启全局共享缓存开关 |
